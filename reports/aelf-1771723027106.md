## 한줄 요약
Aelf v1.11.0 introduces multi‑transaction support, new protobuf structures and several workflow updates. The change mainly adds a `MultiTransaction` type that bundles multiple signed transactions with an optional chain id, and exposes new API endpoints for submitting such batches. The update also brings minor documentation and CI changes.

## 운영자 중요 체크

## RPC/API 영향
- 멀티 트랜잭션을 포함한 블록 조회 시 `TransactionAndChainId` 를 파싱하도록 수정

## 아카이브 노드 영향
- 멀티 트랜잭션이 포함된 블록은 한 블록에 여러 개의 서명된 트랜잭션을 포함하므로, 블록당 데이터 크기가 증가. 아카이브 노드는 전체 체인 데이터를 저장하므로 스토리지 사용량이 증가할 수 있다. 필요 시 압축 옵션 검토.
- 블록 파싱 로직이 `TransactionAndChainId` 를 인식하도록 업데이트 필요. 기존 인덱스 구조가 변동될 수 있으므로 재인덱싱이 필요하다.

## 운영 액션 아이템
- 노드 소프트웨어 업그레이드
- 모니터링 지표 업데이트

## 마이그레이션 체크리스트
- 프로토콜 버전 확인
- 클라이언트 SDK 업데이트
- 멀티 트랜잭션 테스트
- 아카이브 노드 재인덱싱

## 위험/주의 사항
- 기존 RPC 호출에서 `Transaction` 객체를 단일로 받던 부분이 이제 배열 형태(`MultiTransaction`) 로 바뀝. 기존 클라이언트가 새 구조를 인식하지 못하면 호출 실패 위험이 있다.
- 멀티 트랜잭션은 한 블록에 여러 개의 서명된 트랜잭션을 포함할 수 있어, 블록 크기가 커질 가능성이 있다. 아카이브 노드가 전체 체인 데이터를 저장하므로, 블록당 데이터 증가 시 스토리지 사용량이 상승할 수 있다.

## 근거(Evidence)
- 근거 데이터 없음

## 운영 메모
- 새로운 `sendMultiTransaction` 엔드포인트가 추가되며, 요청 바디는 `{ "transactions": [ {"transaction":..., "chain_id":...}, ... ], "signature":... }` 형태를 사용. 기존 단일 트랜잭션 호출과 호환성을 위해 별도 파라미터를 유지.
