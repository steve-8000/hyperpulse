## 한줄 요약
버전 2.0.1 릴리즈는 REST API에서 GZIP 압축 미들웨어를 선택적으로 활성화할 수 있는 옵션을 추가하고, /api/core/v0/* 및 /api/core/v1/* 경로를 공개 라우트에 포함시켜 API 접근 범위를 확장한 변경사항이다. 주요 포인트는 압축 옵션 도입과 라우트 목록 업데이트이며, 버전 번호가 2.0.1 로 상승했다는 점이다.

## 운영자 중요 체크
- 외부 API 엔드포인트/버전 관련 변경 단서가 감지되었습니다.
- 아카이브 노드 운영(보관/색인/동기화) 관련 변경 단서가 감지되었습니다.
- 노드 설정값/프로토콜 플래그 관련 변경 단서가 감지되었습니다.

## 주요 변경점
- 주요 변경점 없음

## RPC/API 영향
- 해당 사항 없음

## 아카이브 노드 영향
- 아카이브 노드가 REST API를 통해 데이터를 수집할 때, GZIP 옵션이 활성화되면 응답 압축으로 인한 CPU 부하가 증가할 수 있다. 대용량 데이터 전송 시 압축 비활성화 옵션을 고려해 주기적 모니터링 필요.
- 공개 라우트에 /api/core/v0/*, /api/core/v1/* 가 추가되었으므로 아카이브 노드가 해당 경로를 호출할 때 인증 필요 여부를 재검토해야 한다.

## 운영 액션 아이템
- config 파일에 "useGZIP": true 가 있는지 확인하고, 필요 시 false 로 설정
- 서버 재시작 후 GZIP 미들웨어가 정상 동작하는지 테스트
- 공개 라우트에 새 경로가 추가되었으므로 방화벽/보안 정책을 재검토
- 대용량 응답 시 CPU 사용량 모니터링

## 마이그레이션 체크리스트
- config_defaults.json 에 "useGZIP": true 가 있는지 확인
- "UseGZIP" 옵션이 명시되지 않은 경우 기본값 true 인지 검증
- 새로운 공개 라우트(/api/core/v0/*, /api/core/v1/*) 가 기존 인증 정책에 부합하는지 테스트
- GZIP 미들웨어가 활성화된 상태에서 클라이언트(예: curl, Postman) 가 정상 응답을 받는지 확인
- 아카이브 노드가 새 라우트에 접근 시 인증 토큰 필요 여부를 재검토

## 위험/주의 사항
- GZIP 미들웨어를 활성화하면 응답 크기가 줄어 서버 부하가 증가할 수 있다. 특히 대용량 파일(예: 블록 데이터) 전송 시 CPU 사용량이 급증할 위험이 있다.
- 공개 라우트에 /api/core/v0/*, /api/core/v1/* 가 추가되면서 인증이 필요 없는 경로가 늘어 보안 취약점이 발생할 가능성이 있다.
- 새 옵션 "UseGZIP" 이 기본값 true 로 설정되었으나, 기존 설정 파일에 명시되지 않으면 자동으로 압축이 적용된다. 이는 클라이언트가 GZIP 지원하지 않을 경우 400 에러를 발생시킬 수 있다.

## 운영 메모
- GZIP 미들웨어가 활성화되면 모든 HTTP 응답이 압축된다. 이는 대용량 데이터(블록, 트랜잭션) 전송 시 CPU 부하를 증가시킬 수 있다. 테스트 환경에서 압축률과 CPU 사용량을 모니터링 할 필요가 있다.
- 공개 라우트에 /api/core/v0/*, /api/core/v1/* 가 추가되었으므로 인증이 필요 없는 경로가 늘어 보안 위험을 주의미한다. 라우트 필터링이 제대로 동작하는지 검증 필요.
- 새 옵션 "UseGZIP" 은 config 파일에 명시되지 않으면 true 로 동작한다. 기존 설정이 없던 경우 자동으로 압축이 적용되므로, 클라이언트가 GZIP 미지원 시 400 에러 발생 여부를 확인해야 한다.
