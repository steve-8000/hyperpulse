## 한줄 요약
본 커밋은 Sui 프로토콜 버전 112에서 메인넷에 ‘address_aliases’ 기능 플래그를 활성화한다. 주소 별칭(주소 별명칭) 사용이 가능해지며, 이는 트랜잭션 서명 검증과 주소 파싱 로직에 영향을 미친다. 현재 RPC/JSON‑RPC 인터페이스에는 직접적인 변경이 없으나, 노드가 새 기능을 인식하도록 설정을 조정해야 한다.

## 운영자 중요 체크
- 아카이브 노드 운영(보관/색인/동기화) 관련 변경 단서가 감지되었습니다.
- 노드 설정값/프로토콜 플래그 관련 변경 단서가 감지되었습니다.

## 주요 변경점
- 주요 변경점 없음

## RPC/API 영향
- 해당 사항 없음

## 아카이브 노드 영향
- 아카이브 노드가 과거 블록을 재인덱스할 때 별칭 주소를 인식하도록 로직이 필요. 현재는 별칭 플래그가 켜져 있지 않으므로 재인덱스 시 별칭이 포함된 주소가 등장하면 인식 실패 가능성.
- 아카이브 노드가 별칭을 지원하도록 업데이트 후 재인덱스(혹은 재시작) 필요.

## 운영 액션 아이템
- 프로토콜 설정 파일에 address_aliases 플래그를 활성화(기본값은 true).
- 노드 재시작 후 별칭 주소가 정상 인식되는지 모니터링.
- 클라이언트(예: explorer, wallet)에서 별칭 주소 지원 여부를 확인.

## 마이그레이션 체크리스트
- 프로토콜 버전 112로 업그레이드 전 테스트넷에서 별칭 주소 지원 테스트.
- 노드 설정 파일에 address_aliases 플래그가 true인지 확인.
- 아카이브 노드 재시작 후 인덱스가 정상 동작하는지 검증.
- RPC/JSON‑RPC 호출 시 별칭 주소를 전달해 정상 응답이 나오는지 테스트.

## 위험/주의 사항
- 주소 별칭 기능이 활성화되면 기존 주소 포맷(32‑바이트)과 별칭 문자열 간 변환 로직이 필요해져, 이 과정에서 잘못된 파싱은 트랜잭션 거부로 이어질 수 있다.
- 구버 노드(아카이브)에서 별칭 주소를 인식하지 못하면 과거 데이터와 새 데이터 간 불일치가 발생해, 재인덱스 필요성 있음.
- 프로토콜 버전 112로 업그레이드 시, 별칭 지원이 없던 클라이언트는 예외를 발생시킬 수 있다.

## 운영 메모
- 프로토콜 설정에 address_aliases 플래그가 추가되었으며, 이는 메인넷에서 별칭 주소를 허용한다는 의미다.
- RPC/JSON‑RPC 레이어에는 별칭을 인식하도록 파라미터가 추가되지 않았으므로, 기존 RPC 엔드포인트는 그대로 사용 가능.
- 노드가 별칭을 인식하려면 프로토콜 설정 파일에 플래그를 활성화하고 재시작해야 함.
