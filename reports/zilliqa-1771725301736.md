# Zilliqa 운영 리포트

## 한줄 요약
본 릴리즈는 Zilliqa 메인넷에서 이더리움(eth) 로그 저장 정책을 2일에서 약 7일로 늘린 업데이트이며, 버전 태그를 v9.3.4 → v9.3.6 으로 변경한 소규모 수정이다. 주요 파일은 `src/libEth/filters/APICache.cpp`와 `src/libUtils/SWInfo.h`이며, 로그 저장 깊이 상수값(`TXMETADATADEPTH`)을 2880에서 17150(≈7일) 으로 조정했다. 이 변경은 RPC/JSON‑RPC 인터페이스에는 직접적인 변동이 없으나, 로그 저장 정책이 바뀌면서 아카이브 노드의 스토리지 요구량과 필터 캐시 동작에 영향을 미친다.

## 운영자 중요 체크
- 아카이브 노드 운영(보관/색인/동기화) 관련 변경 단서가 감지되었습니다.

## 주요 변경점
- eth 로그 저장 기간을 약 2일 → 7일 로 연장
- 버전 태그 업데이트

## RPC/API 영향
- 해당 사항 없음

## 아카이브 노드 영향
- 로그 저장 깊이 증가로 인한 디스크 사용량 급증(약 3~5배)
- 필터 캐시가 오래된 로그를 보관해야 하므로 메모리/디스크 I/O 부하 증가
- 필터 API 호출 시 오래된 로그를 탐색하는 비용이 상승

## 운영 액션 아이템
- 노드 설정 파일(`config.xml` 등)에서 `TXMETADATADEPTH` 값이 17150 으로 반영되었는지 확인
- 스토리지 용량을 재평가하고 필요 시 디스크 확장 계획 수립
- 필터 캐시 크기(`filters/size`)를 조정해 메모리 사용량을 모니터링
- 업그레이드 후 로그 저장 기간이 7일로 늘어도, 오래된 로그를 정리하는 스크립트가 정상 동작하는지 테스트

## 마이그레이션 체크리스트
- 노드 소프트웨어를 최신 버전(v9.3.6) 으로 업그레이드
- 필터 캐시 설정(`TXMETADATADEPTH`)이 17150 으로 반영되었는지 확인
- 스토리지 용량 및 백업 정책(아카이브 노드) 재검토
- 필터 API 호출 시 오래된 로그를 탐색하는 테스트 수행
- 버전 태그(`VERSION_TAG`)가 클라이언트에 반영되었는지 확인

## 위험/주의 사항
- 로그 저장 깊이 증가로 인한 스토리지 사용량 급증(특히 대규모 노드에서 3‑5배 증가 가능)
- 필터 캐시가 오래된 로그를 더 많이 보관해야 하므로 메모리/디스크 I/O 부하가 증가할 수 있음
- 필터 API 호출 시 오래된 로그를 탐색하는 비용이 증가해 응답 지연 가능성
- 버전 업그레이드 시 기존 노드가 새 설정값을 인식하지 못하면 로그 불일관성 문제 발생

## 근거(Evidence)
- 근거 데이터 없음

## 운영 메모
- 필터 캐시가 오래된 로그를 보관하는 범위가 2일에서 7일로 확대되었으므로, 필터 API 호출 시 오래된 로그를 검색하는 비용이 증가한다.
- 아카이브 노드가 모든 블록을 보관하므로, 로그 저장 깊이를 늘리면 디스크 사용량이 3~5배 증가할 수 있다. 이는 스토리지 비용과 재동기화 주기에 직접적인 영향이 있다.
- 버전 태그 변경은 RPC/JSON‑RPC 인터페이스에 직접적인 변동이 없으나, 클라이언트가 버전 정보를 확인할 때 사용되는 `VERSION_TAG` 값이 바뀌므로, 클라이언트가 이 값을 참조하는 경우 업데이트 필요.
