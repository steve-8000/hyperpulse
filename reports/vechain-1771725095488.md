## 한줄 요약
VeChain Thors v2.4.2‑rc1은 ABI 패키지 재구성, 리프로세스 기능 추가, 트랜잭션 서명 검증 강화 등 다수의 핵심 변경을 포함한다. 주요 포인트는 ABI 구현이 외부 패키지에서 내부 구현으로 이동하면서 JSON‑RPC 인풋/아웃풋 처리 로직이 바뀌고, 트랜잭션 서명 검증 로직이 tx‑pool 단계에서만 low‑S를 강제한다는 점이다. 또한 Dockerfile에 2112 포트가 추가되었으며, 리프로세스 기능이 노드 재구동 시 모든 트랜잭션과 블록을 다시 처리하도록 한다. 이로 인한 운영 리스크와 마이그레이션 포인트를 정리한다.

## 운영자 중요 체크
- 외부 API 엔드포인트/버전 관련 변경 단서가 감지되었습니다.
- 아카이브 노드 운영(보관/색인/동기화) 관련 변경 단서가 감지되었습니다.
- 노드 설정값/프로토콜 플래그 관련 변경 단서가 감지되었습니다.

## 주요 변경점
- 1. ABI 패키지(abi/ethabi) 새로 추가, 기존 go‑ethereum/abi에서 내부 구현으로 교체
- 2. Dockerfile에 2112 포트 추가(기존 8669,11235,55555/udp)
- 3. reprocess 명령어 추가(cmd/thor/reprocess/reprocess.go)
- 4. tx‑pool에서 low‑S 서명 검증 강화(#1575)
- 5. metrics/프로세스 메트릭스 추가 및 리소스 사용량 모니터링 개선

## RPC/API 영향
- ABI 인풋/아웃풔 파싱 로직 변경(내부 패키지 교체)
- reprocess 명령어 추가

## 아카이브 노드 영향
- 1. reprocess 기능이 아카이브 노드에서 트랜잭션 재처리를 요구할 수 있다. 이때 메모리/CPU 사용량이 급증한다
- 2. Dockerfile에 2112 포트 추가로 인한 네트워크 설정 변경 필요

## 운영 액션 아이템
- 1. 재시작 전 리소스 모니터링(메모리/CPU) 설정
- 2. reprocess 기능이 필요 시 명령어 실행 여부 검증
- 3. ABI 파싱 로직 변경에 따른 클라이언트 테스트

## 마이그레이션 체크리스트
- 1. ABI 패키지 교체에 따른 JSON‑RPC 인풋/아웃풔 파싱 로직 테스트
- 2. reprocess 기능이 활성화된 노드에서 재시작 시 전체 트랜잭션 재처리 여부 확인
- 3. Dockerfile 포트 2112가 외부에 노출되는지 검증

## 위험/주의 사항
- 1. ABI 패키지 재구성으로 인한 JSON‑RPC 인풋/아웃풋 파싱 로직이 변경되어 기존 클라이언트가 ABI를 직접 호출할 경우 호환성 이슈 발생 가능
- 2. reprocess 기능이 추가되면서 노드 재시작 시 모든 트랜잭션을 다시 처리한다. 이는 대용량 노드(아카이브)에서 재시작 시 장시간과 메모리 사용량이 급증할 수 있다
- 3. low‑S 서명 검증 로직이 tx‑pool 단계에서만 적용되면서, 기존에 허용됐던 서명(높은 S 값) 트랜잭션이 거부될 수 있다

## 운영 메모
- - ABI 구현이 내부 패키지로 이동하면서 JSON‑RPC에서 `eth_call` 시 인풋/아웃풋 파싱 로직이 바뀌므로, 기존에 `eth_call`을 직접 호출하던 클라이언트는 ABI 파싱 로직이 달라서 결과가 달라질 수 있다
- - reprocess 기능은 모든 트랜잭션과 블록을 재처리한다. 아카이브 노드(보관)에서는 이 기능이 비활성화 상태에서만 동작하도록 설계되었으나, 재시작 시 전체 트랜잭션을 다시 처리하면 메모리/CPU 부하가 급증한다
- - Dockerfile에 2112 포트 추가는 RPC/CLI에서 새 포트를 노출한다. 기존 클라이언트가 2112를 사용하도록 설정해야 한다
