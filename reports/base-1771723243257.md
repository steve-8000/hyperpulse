## 한줄 요약
본 릴리즈는 Base 노드가 의존하는 Reth Node 버전을 0.3.1에서 0.4.0으로 업그레이드합니다. 버전 변경은 `versions.env`와 `versions.json`에 반영되었으며, 실제 코드 내부에는 API/JSON‑RPC 변경이 없습니다. 다만, Reth 0.4.0의 내부 구현 변화가 노드 동작에 영향을 미칠 수 있으므로 운영자는 주의 검토를 권장합니다.

## 운영자 중요 체크
- RPC/JSON-RPC 관련 변경 단서가 감지되었습니다.
- 아카이브 노드 운영(보관/색인/동기화) 관련 변경 단서가 감지되었습니다.

## 주요 변경점
- 주요 변경점 없음

## RPC/API 영향
- 해당 사항 없음

## 아카이브 노드 영향
- Reth 0.4.0이 내부 스토리지 포맷을 바뀜 경우, 아카이브 노드가 기존 블록 데이터를 재인덱스할 필요가 있을 수 있습니다.
- 아카이브 노드의 재시작 시, `reth-node` 버전 차이에 따라 데이터베이스 스키마가 변동될 수 있으니, 재시작 전 백업을 권장합니다.

## 운영 액션 아이템
- go.mod 및 go.sum 파일을 최신화하고 `go mod tidy` 실행
- Reth 0.4.0에 대한 테스트 환경에서 JSON‑RPC 엔드포인트(예: `/eth/txpool`, `/eth/blockByNumber`)를 검증
- 아카이브 노드가 운영 중이라면, 재시작 전 데이터베이스 백업 수행
- 배포 후 모니터링 지표(블록 수신률, RPC 응답시간) 확인

## 마이그레이션 체크리스트
- go.mod/dep 업데이트 후 `go mod vendor` 혹은 `go get -u ./...` 실행
- 테스트넷에서 0.4.0 버전으로 테스트 후 메인넷 배포
- JSON‑RPC 스키마 호환성 검증(예: `eth_getBlockByNumber` 반환값)
- 아카이브 노드 재시작 시 스냅샷 복구/재인덱스 테스트
- 모니터링 지표(블록 수신률, RPC 응답시간) 24h 이상 모니터링

## 위험/주의 사항
- Reth 0.4.0이 기존 0.3.x와 비교해 내부 구조(예: 블록 파싱, 상태 저장 방식)에서 변경이 있을 수 있어, 현재 노드가 의존하는 서브 모듈(예: `reth-node` 내부 RPC 엔드포인트)에서 비호환성이 발생할 가능성이 있습니다.
- 업그레이드 시, 기존에 사용하던 JSON‑RPC 스키마가 변동될 경우, 외부 클라이언트(예: 지갑, 탐색기)에서 예외가 발생할 수 있습니다.
- 버전 업그레이드에 따라, 테스트 환경에서 충분히 검증되지 않은 상태라 운영 중 장애 발생 위험이 존재합니다.

## 근거(Evidence)
- 근거 데이터 없음

## 운영 메모
- Reth Node 버전이 바뀜 만큼, 해당 리포지터의 `go.mod` 혹은 의존성 관리 파일이 자동으로 업데이트되지 않을 수 있으니, `go mod tidy`를 실행해 의존성 충돌을 확인해야 합니다.
- Reth 0.4.0이 새로 도입한 `reth-node` 내부 API가 기존에 사용하던 RPC 엔드포인트(예: `/eth/txpool`)와 호환되는지 테스트가 필요합니다.
- 아카이브 노드(보관 정책)에서는 블록 저장 포맷이 바뀜 경우, 스토리지 구조가 변동될 수 있으니, 현재 운영 중인 아카이브 노드의 스냅샷을 백업 후 재시작 테스트를 권장합니다.
